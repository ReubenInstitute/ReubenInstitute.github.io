# 19 Aug 2025 – DeepSeek 3 – Parashot system and dual inheritance implementation

_Code_

With the architecture and parsing specification solid, we turned to the most structurally complex content type: Parashot.  The session’s lasting contribution was not just the code, but a design pattern that would shape the entire platform.

### The dual inheritance pattern

We introduced two base classes and let content types pick one or both, depending on their role:
* `MarkdownSource` – for anything backed by a user‑editable Markdown file.
* `BaseContent` – for anything that will eventually generate media and output.

This gave us a clean, scalable mix‑in model:
* `Psalm` and `BibleArticle` inherit from **both** – they are editable *and* media‑generating.
* `Parashah` inherits only from `MarkdownSource` – it’s a container, not a media unit.
* `ParashahEpisode` inherits only from `BaseContent` – it’s the actual media‑producing piece inside a Parashah.

The pattern avoided forcing every class into the same mould while keeping the API consistent across the platform.

### Content classes implemented

**`Psalm`** (`MarkdownSource + BaseContent`)
* User‑editable `psalms/001.md` files.
* Future media generation capability.
* Clean verse parsing with `\n\n` breaks.

**`BibleArticle`** (`MarkdownSource + BaseContent`)
* Curated Bible passages with verse numbering.
* `articles/daniel-dream.md` user customisation.
* Segment‑and‑verse hierarchy parsing.

**`Parashah`** (`MarkdownSource` only)
* Container for Torah portions with CSV‑driven episodes.
* `parashot/01.md` + `parashot/01.csv` structure.
* Triple newline episode breaks (`\n\n\n`).

**`ParashahEpisode`** (`BaseContent` only)
* The actual media‑generating unit inside a Parashah.
* 3‑level hierarchy parsing: episode → segment → verse.
* Verse reference parsing via `chapter.verse` pattern.

### Critical technical solutions

Four problems were solved during the implementation:
1. **Initialisation order** – attributes must be set *before* parent `__init__` calls, otherwise the hierarchy collapses.
2. **CSV string handling** – proper dtype conversion for verse references, preventing silent misreads.
3. **File creation logic** – Markdown files are auto‑generated from the Bible source whenever they’re missing.
4. **Hierarchical parsing** – consistent rules across all content types, regardless of depth, so the same parser logic works everywhere.

### File structure

```
psalms/001.md          # Psalm markdown files
articles/daniel-dream.md # Article markdown files
parashot/01.csv        # Episode definitions
parashot/01.md         # Parashah text with episode breaks
```

### Working system features

* Automatic file creation when missing.
* User responsibility model – CSV defines structure, user maintains consistency.
* Progressive complexity – empty CSV = one episode, custom CSV = multiple episodes.
* Clean 0‑based indexing throughout.
* Comprehensive error handling and debugging support.

### Philosophical backbone

* Minimal markup – uses natural text structure.
* User empowerment – full control via editable files.
* Sacred text respect – careful structural preservation.
* Future‑proof – easy to add new content types.
