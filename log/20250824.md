# 24 Aug 2025 – DeepSeek 3 – Markdown API and the super‑zombie bug

_Code_

The session set out to “clean to death” the `MarkdownSource` class and emerge with a robust, reusable Markdown API that every scripture class could share.  The design work was meticulous.  The debugging that followed was legendary.

### The architectural decisions

Three principles were locked in after extended debate:

#### The “Styling Overlay” concept

Markdown files are not a replacement for the canonical Hebrew text.  They are a non‑destructive formatting layer—modern punctuation, spacing, line breaks—applied *over* the immutable canonical JSON.  The source remains untouched; the overlay is pure presentation.

#### The “Initialize then Update” pattern

Every object always initialises from the canonical JSON first.  Only after it’s a complete, valid object does it attempt to find and apply a Markdown file as an overlay.  If the file doesn’t exist yet, the object is still fully functional.

#### The final API

A clean, three‑part interface was adopted for every markdown‑enabled class:
* `.markdown` (property) – gets the object’s content as a formatted Markdown string.
* `load_markdown()` (public method) – manually reloads and parses content from the `.md` file.
* `set_markdown(text)` (public method) – takes a string, saves it to the `.md` file, and updates the object’s state.

#### A lean `MarkdownSource` mixin

The base class was stripped of all complex logic and reduced to a simple mixin that provides file I/O utilities and marks a class as markdown‑capable.  No more over‑engineering.

### Taming the super‑zombie

The new architecture was implemented across all three core Rabbinic modules: `TalmudBavli.py`, `TalmudYerushalmi.py`, and `Mishnah.py`.  The first two went smoothly.  `Mishnah.py` did not.

A “super‑zombie” bug had been lurking for sessions—something deep in the parsing kept resurrecting errors that seemed fixed.  We attacked it with a dedicated diagnostic test, and the root cause finally surfaced: the `MishnahVerse` class was an unnecessary layer.  The Mishnah’s data structure is different from the Talmuds, and treating a Mishnaya as a single paragraph—rather than a collection of verse objects—was the correct model.

Removing `MishnahVerse` killed the bug.  The Mishnaya became a self‑contained paragraph with double newlines (`\n\n`) separating its internal verses, exactly matching how users would edit them.  The three modules were finally polished, debugged, and consistent.

The session ended with a clean, reusable Markdown API and a platform that had finally exorcised its most persistent ghost. 🧟
