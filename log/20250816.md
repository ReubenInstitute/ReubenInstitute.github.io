# 16 Aug 2025 – DeepSeek 3 – Text parsing specification for Psalms, Articles, and Parashot

_Code_

The session defined exactly how the platform’s three content types would be parsed from raw Markdown into structured data — a specification that had to honour the sacred texts while being machine‑readable and round‑trippable.

### The core challenge

We needed to parse three different text structures using minimal markup, preserving content integrity and enabling round‑trip regeneration.  The hierarchy was the key:
* Psalms — 1 level (verses only)
* Articles — 2 levels (segments containing verses)
* Parashot — 3 levels (episodes containing segments containing verses)

### The parsing specification

#### Blank line definitions
* Single blank line: `\n\n` (two newlines) — separates verses.
* Double blank line: `\n\n\n` (three newlines) — separates segments or episodes.

#### Content‑specific rules

**Psalms (1‑level hierarchy)**
* Verse breaks: single blank lines (`\n\n`).
* Segments: each verse becomes one segment.
* Numbering: implicit (1, 2, 3…).

**Articles (2‑level hierarchy)**
* Verse breaks: single blank lines (`\n\n`).
* Segment breaks: double blank lines (`\n\n\n`).
* Numbering: none — articles have no verse numbers in the source.

**Parashot (3‑level hierarchy)**
* Verse breaks: explicit numbering pattern (`chapter.verse`).
* Segment breaks: single blank lines (`\n\n`).
* Episode breaks: double blank lines (`\n\n\n`).

#### Verse text processing rules
During parsing, three operations are applied to every verse:
1. Strip trailing newline — it’s a structural artifact, not content.
2. Preserve internal newlines — meaningful line breaks within a verse.
3. Extract verse numbers — stored as metadata, not in the text content.

### The unified API

All content objects expose the same interface:
* `.segments` → list of segments.
* Each segment contains `.verses` → list of verses.

### Design principles achieved
* Minimal markup — uses existing text structure, no new syntax.
* No ambiguity — clear hierarchical breaks prevent parsing conflicts.
* Content integrity — original formatting and structure are preserved.
* LLM‑friendly — natural reading flow remains intact for AI processing.
* Future‑proof — easy to extend with new content types.

The specification gave the entire platform a single, unambiguous contract for reading and writing sacred text — the backbone everything else would build on.
