In this session, we analyzed the architectural inconsistencies in our media processing pipeline and mapped out a strategy to move from a rigid, "bossy" hierarchy to a clean, object-oriented structure.

The Core Problem
We identified that our filenames and paths are currently "spread all over the code" and inconsistent, with different classes independently guessing how to format strings like book numbers (02d vs 03d). Furthermore, the BaseVideo and BaseAudio classes were identified as "crap" because they forced a fixed sequence of events (like alternating video/audio streams) that broke when custom logic, such as title cards or background music, was introduced.

The Architectural Shift: Pure Object Orientation
To fix this, we agreed on a "Single Source of Truth" approach where the artifact object owns its own metadata:

· Artifacts Own Paths: Instead of a "Manager" class, each object (like a PsalmSlide or PsalmHebrewVideo) is responsible for its own file path.
· Composition over Inheritance: We decided to gut the base classes. The derived classes (like PsalmHebrewVideo) now take 100% responsibility for assembling their own internal timelines and streams.
· Base Classes as "Render Engines": The base classes were reduced to "dumb printing presses" that only handle the final ffmpeg execution with standardized encoding settings (codecs, bitrates, etc.).

Implementation Steps Taken
We defined the specific properties needed to standardize the interface between derived and base classes:

· Standardized Streams: Added video_stream and audio_stream properties to PsalmHebrewVideo to return finalized ffmpeg nodes, pulling the assembly and mixing logic out of the generation function.
· Clean Exporting: Defined a new export function for BaseVideo that replaces the old generate logic. It treats the streams as "black boxes" and simply renders them to the location provided by the object.
· Path Consistency: Moving forward, the filename and folder logic will be encapsulated within the classes themselves to eliminate reliance on global folder variables and duplicated string formatting.

## 17 Mar 2026 – Gemini 3 Pro

This session was a deep dive into an obscure architectural flaw in the ffmpeg-python library that was causing our video renders to crash. We moved from diagnosing a cryptic error to identifying a physical limitation of MP3 encoding, and finally to implementing a "cache-busting" architectural fix.

1. The Diagnosis: Invisible Graph Merging
The session began with the error ValueError: Encountered format('rgba') ... with multiple outgoing edges.
We discovered that ffmpeg-python tries to be "smart" by optimizing our filter graph. If two different verses in our loop use the same background image and have the exact same duration, the library assumes they are the same node and merges them into one. When our code then tries to apply a "Fade In" to one verse but not the other, the graph branches. FFmpeg forbids this branching without an explicit split filter, causing the crash.

2. The Discovery: MP3 Frame Quantization
We suspected the durations in our cloned.csv were being quantized. By cross-referencing our source audio with the ElevenLabs output, we proved our theory:

· The Cause: ElevenLabs generates MP3s in discrete chunks called "frames."
· The Result: If two original verses have similar lengths (within ~0.026 seconds), the MP3 encoder rounds them to the exact same number of frames.
· The Trigger: This produces identical file sizes and identical duration floats (e.g., 10.448937), which triggers the ffmpeg-python hash collision.

3. The Clean-Up: Decoupling Streams
We refactored the ParashahEpisodeVideo and ParashahVideo classes to simplify the rendering logic:

· Separation of Concerns: We replaced the complex segments property with independent video_stream and audio_stream properties.
· Removing Redundancy: We realized that since ElevenLabs provides the correct length, we could remove the .audio.filter('atrim') entirely, letting FFmpeg simply read until the End-of-File (EOF).
· Streamlining: We commented out unnecessary safety checks (if not verse_obj, etc.) to make the code more readable and maintainable.

4. The Solution: Global Cache Busting
The "Gold Medal" fix was moved out of the video logic and into the Data Layer within AudioAI.py.
Instead of complicated logic in the render loop, we implemented a Cache Buster in the CSV loader. When the class loads durations into memory, it tracks them in a set. If it encounters a duplicate duration, it subtracts a single microsecond (0.000001) until the value is unique.

Why this works perfectly:

· Mathematically Unique: Every duration in memory is now a unique "hash," so the library never attempts to merge nodes.
· Physically Imperceptible: A microsecond difference is invisible to the human eye and ear.
· Non-Destructive: The actual cloned.csv on our disk remains unchanged and accurate.

By the end of the session, we successfully bypassed a deep-seated library bug with a single, elegant line of data-layer code, enabling us to render anything from a single episode to the entire Bible without a single graph conflict.
