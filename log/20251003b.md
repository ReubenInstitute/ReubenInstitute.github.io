# 3 Oct 2025 – DeepSeek 3 – Audio system with CSV source of truth

_Code_

The audio architecture we'd built was broken in ways that only became visible when you tried to scale it.  Circular imports between `Bible` and `Audio` classes made the import graph a fragile house of cards.  A full scan of 23 000 verses took 1.6 minutes — every time — because the system was checking the file system for every single file rather than consulting a single source of truth.  Java‑style getter patterns and inconsistent API design added a layer of cognitive friction to every interaction.

### The fix: CSV as the database

The breakthrough was recognising that the CSV files *are* the database.  File‑existence checks were unnecessary.  We moved audio data to the Bible scope, not the verse level, and switched to in‑memory dictionary lookups.  `BibleAudio` now handles file operations; `BibleAudioAI` handles AI generation.  The API became Pythonic — `bible.audio.has_original_audio(verse)` instead of awkward verse‑level audio objects.

The performance impact was immediate: a scan that previously burned 116 030 file‑system operations now completes in seconds.  Working audio statistics show 2 740 original verses (11.8%) and 528 cloned verses (2.3%).  An AI consistency analysis revealed 99.8% duration consistency across the cloned audio — a number that validated the entire cloning pipeline.

### Coding standards formalised

In parallel, we locked down nine coding standards that would govern every line from this point forward:
* Tabs only for indentation — 1 byte, not 4, and semantically correct.
* Comments only when code is unclear — no obvious statements.
* No blank lines inside functions; refactor if they feel necessary.
* Full variable names — `verse_number`, never `verse_num`.
* Specific exceptions only — no bare `except:`.
* Pragmatic line length — no strict 79‑character limit.
* Double quotes for user‑facing strings, single for internal code.
* One method per line when chaining, for clarity.
* Absolute imports preferred, with stdlib / third‑party / local grouping.

These weren't arbitrary rules plucked from a style guide.  They were the distilled response to every pain point we'd hit in the previous sessions — the frictional APIs, the cryptic variable names, the functions bloated with blank lines and catch‑all exception handlers.

The audio system was now production‑ready, integrated with the Bible core, and the statistics were fast and accurate.  The next step was clear: functional API refactoring.
