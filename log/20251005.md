# 5 Oct 2025 – DeepSeek 3 – Text system refactoring with TextStore

_Code_

The Markdown‑based text system we’d built earlier was functional, but it had a fundamental flaw: thousands of independent file checks every time the platform loaded.  Each Psalm, each article, each parashah was a separate disk access, and the cost added up fast — about 54 seconds for a full load.  This session redesigned the entire text subsystem around a singleton storage manager that reads once and serves everywhere.

### Core architecture

Four classes now handle the entire text pipeline:

* **`TextStore`** — a singleton storage manager that scans the file system once, caches every text file in memory, and serves content to every object that needs it.  No more scatter‑shot file I/O.
* **`Text`** — the base class that any content object can extend.  It automatically retrieves its text from `TextStore` without knowing where the data lives.
* **`PoemText`** — a specialised parser for poetic content, designed for Psalms.  It understands that `\n` means a line break, `\n\n` separates verses, and `\n\n\n` separates paragraphs.
* **`NarrationText`** — a specialised parser for narrative content, designed for Parashot and Articles.  It handles chapter‑and‑verse numbering patterns, multi‑line verses, and paragraph grouping.

### Key technical decisions

The singleton pattern on `TextStore` eliminates dependency injection complexity entirely — any object that needs text can get it without plumbing a reference through the constructor chain.  Both parsers return a unified output structure (`[[[[numbers], lines], …], …]`) so the rest of the platform never has to care which parser produced it.  Storage, parsing, and content logic are cleanly separated — each layer owns exactly one responsibility.

### Parsing specifications finalised

* **PoemText**: `\n` = lines, `\n\n` = verses, `\n\n\n` = paragraphs.
* **NarrationText**: chapter‑and‑verse numbering, multi‑line verse support, paragraph grouping.

### What’s ready for implementation

The next step is straightforward: update `Psalm` to inherit from `PoemText`, update `Parashah` to inherit from `NarrationText`, create the `PoemArticle`/`NarrationArticle` hierarchy for articles, and remove the legacy `MarkdownSource` mixin from the Bible classes.  The bulk file scanning alone is expected to drop load time from 54 seconds to roughly 5–8 seconds, and the architecture is clean enough to scale to any future scripture types.
