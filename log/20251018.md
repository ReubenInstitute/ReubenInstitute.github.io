# 18 Oct 2025 – DeepSeek 3 – AudioPlayer class hierarchy and Flask routes

_Studio_

We built a proper audio player system from scratch—no hacks, no global event spaghetti, just a clean class hierarchy and cleanly separated Flask routes.  The result lets us edit verse-level audio with waveform visualisation, region markers, and a professional save workflow, all inside the Production Studio.

### The class architecture

At the foundation is an `AudioPlayer` base class with the universal controls: `play()`, `pause()`, `stop()`, `seek()`, and `playPause()`.  From that base we built three specialised players:
* `SimpleAudioPlayer` – invisible playback, zero UI, pure audio control.
* `VisualAudioPlayer` – waveform visualisation for browsing and listening.
* `MarkerAudioPlayer` – region‑based marking and cutting, the core of the editing workflow.

### MarkerAudioPlayer

The marker player is the workhorse.  Users can drag and resize regions directly on the waveform to define start and end points for audio segments.  Coordinates are converted from display space to actual timing, and form fields update directly—no convoluted event system, no global state, just clean communication between the JavaScript and the HTML.  Playback controls are kept separate from the marker selection logic so the two never interfere.

### Flask integration

The routes follow the platform’s existing URL patterns exactly:
* Psalm audio editing: `/bible/psalms/<psalm>/<verse>/audio`
* Parashah audio editing: `/bible/parashot/<parashah>/<episode>/<verse>/audio`
* Audio segment serving: `/audio-<book>-<frm>-<to>.wav` — powered by FFmpeg on the fly.
* Dedicated save routes for each content type, with proper redirects after save.

### Implementation principles

We aggressively avoided the amateur‑hour patterns that plague so many audio‑editing UIs.  No `setTimeout` or `setInterval` hacks.  No DOM manipulation inside the player classes.  No global event system masquerading as architecture.  Templates stay pure; JavaScript logic lives encapsulated in the player objects; play buttons are injected via proper dependency injection, not jQuery soup.

### User experience

The editing workflow is a two‑button system: **Save** keeps you on the current verse so you can review your edit, and **Save & Next** advances to the next verse for efficient batch processing.  Users can start and stop playback with the play/pause button or by clicking directly on the waveform.  Region handles work on mobile, and navigation back to parent pages is always a single click.

### Key technical decisions

We reused the existing `players` Map instead of polluting the global namespace.  Form updates are direct—no event bus, no pub/sub over‑engineering.  Separate save routes per content type keep the logic clean and make future expansion trivial.  FFmpeg handles audio segment extraction directly, so the waveform display always matches the raw audio data exactly.

The implementation brings the original audio editing functionality into the new studio with professional‑grade code quality and a user experience designed for real production work.
