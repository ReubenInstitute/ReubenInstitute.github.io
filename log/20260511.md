# 11 May 2026 – DeepSeek 3 – PsalmCover layout fixed, extreme‑chars and draw_text bugs killed

_Font, Code, Media, PsalmSeries, Studio_

What began as a routine cover‑generation update spiralled into a six‑hour debugging war that exposed two deeply hidden, interacting bugs in the Hebrew text‑rendering pipeline.  By the end, we had completely stabilised the Psalm cover layout for all three aspect ratios (vertical, horizontal, square), retired the flawed “raw”/“enhanced” naming scheme, and built a clean, deterministic background‑generation pipeline.

### The symptoms

After refactoring `PsalmCover` to use pre‑extracted background JPEGs and a new centred‑square layout model, the vertical covers looked perfect—exactly like the originals—but the horizontal and square covers were vertically misaligned and, worse, psalm numbers that did **not** contain a lamed or qof were being clipped mid‑screen.  The clipping was not at the canvas edge; the glyph itself was cut.

### Bug 1 – bogus extreme‑char heights

The first bug was in `Image._get_extreme_chars()`.  When determining the font’s maximum vertical extent, the function iterated over every Hebrew letter combined with every diacritic and cantillation mark.  For LulavCLM—a font that contains **no** diacritics or cantillation glyphs—these combinations fell back to the `.notdef` (tofu) glyph.  The resulting “extreme” characters (`'שׂ֮'` and `'קְ'`) produced wildly inflated bounding‑box heights: 325 px for the 120 pt title and 702 px for the 260 pt number, where the true heights for the tallest real letters (lamed and qof) were only 152 px and 328 px.

This bogus data had contaminated every layout constant and explained why the vertical cover’s block centre had settled at an impossible 0.565—a ghost value that masked a true visual centre far lower.

**The fix:** We added a `has_glyph()` check using `fonttools`’s `TTFont` cmap, skipping any character combination that doesn’t exist in the font.  The corrected extreme chars for LulavCLM are now simply `'לק'`, and `bbox()` returns physically plausible heights.

### Bug 2 – draw_text Y offset

Even with the corrected heights, non‑lamed numbers remained clipped.  The root cause was in `Image.draw_text()`, in the branch that does **not** use `anchor="mm"`.  The function creates a temporary image sized to the full extreme‑chars height and draws the actual text (plus extreme chars) into it.  However, it drew at `y=0`, whereas the correct position—to align the text’s baseline correctly within the extreme‑height box—is `y = -full_bbox[1]`.  The `anchor="mm"` branch already did this; the non‑anchor branch was missing the offset.

For numbers that contain a lamed, the lamed’s huge ascender made `full_bbox[1]` very negative, over‑compensating for the missing offset and accidentally making them render correctly.  For all other numbers, the text sat too high inside the temporary image, and the bottom of the glyph—and its shadow—extended beyond the canvas, getting clipped.

**The fix (credit to Claude):** Change the draw call from
``python
temp_draw.text((draw_x, 0), text + extreme_chars, …)
``
to
``python
temp_draw.text((draw_x, -full_bbox[1]), text + extreme_chars, …)
``
This one‑character change eliminated the clipping for all psalms and made the horizontal and square covers snap to perfect vertical centering.

### PsalmCover constants recalculated

With both bugs fixed, we derived the true visual centres of the original vertical cover:

- Title visual centre: **0.382** of the canvas height (not 0.45)
- Number visual centre: **0.534** (not 0.68)
- Block visual centre: **0.458** (below the canvas centre, as suspected)

From these we computed the final constants inside the centred‑square layout model, which are now locked in `PsalmCover`:

``python
TITLE_FONT      = 0.167
NUMBER_FONT     = 0.361
TITLE_Y         = 0.3646
NUMBER_Y        = 0.6354
SAFEZONE_CENTER = 0.458    # vertical cover block centre
``

- Vertical covers reproduce the original safe‑zone lift perfectly.
- Horizontal and square covers are now mathematically centred at exactly 0.5.
- No clipping on any psalm, regardless of letter content.
- A square cover size constant `SD = (1000, 1000)` was added to `Media.py`.

### New routes and folder structure

The old `-raw`/`-bare` suffix system was retired.  The final output folder structure is:

```

psalms/                  # Library (pure Shmuelov, fog only)
├── psalm001.mp3
├── psalm001.mp4
└── covers/
├── psalm001.jpg
├── square/psalm001.jpg
└── horizontal/psalm001.jpg

psalms+/                 # Psalms Plus (creative series)
├── psalm001.mp3         (Score audio)
├── score/               (fog + music video)
├── cinematic/           (custom graphics + music video)
└── covers/              (cinematic covers)

```

Two routes were added to `studio.py`:

- `/generate/covers/psalm/<int:number>` – generates all three covers for a single psalm.
- `/generate/covers/all` – generates all three covers for all 150 psalms.

A `Generate All Covers` button links to the bulk route from the main Psalms page.

### Background generation script

`generate-backgrounds.py` was rewritten as an idempotent, source‑agnostic script.  From the original `back.mov` it produces:

- `back.mp4` – centred square, desaturated, scaled by 0.9 (for the square cover).
- `back-720x1280.mp4` / `.jpg` – vertical fog loop and its cover frame.
- `back-1280x720.mp4` / `.jpg` – horizontal fog loop and its cover frame.

It also scans `assets/` for psalm‑specific cinematic background MP4s and extracts missing `.jpg` frames (at 2 s) if they don’t exist.  An `--overwrite` flag forces regeneration.

### State at session end

The entire cover pipeline is now deterministic, pixel‑perfect, and free of the ghost values that had plagued every layout attempt.  The vertical covers—the only ones with a 7‑day change limit on TikTok—are perfect.  The horizontal and square covers are correctly centred and can be refined further on YouTube without any platform deadline.  The naming crisis is over; the folder structure is the documentation.
