# 17 Mar 2026 – Gemini 3 Pro – ffmpeg‑python MP3 frame bug solved, global cache buster

_Code_

A cryptic FFmpeg error had been blocking complex video renders for weeks: `ValueError: Encountered format('rgba') … with multiple outgoing edges`.  This session traced the bug to its root cause—a physical limitation of MP3 encoding—and fixed it with a single, elegant line of data‑layer code.

### The diagnosis: invisible graph merging

The error surfaced whenever we tried to render a Parashah video that concatenated multiple episodes.  Each verse stream uses the same background image, and `ffmpeg-python` tries to be smart about its filter graph: if two streams share the same source and have identical durations, the library merges them into a single node.  When our code then applies a Fade In to one verse but not the other, the graph branches.  FFmpeg forbids branching without an explicit `split` filter, so the render crashes.

The question was *why* different verses were ending up with identical durations in the first place.  They shouldn't have been: the cloned audio was generated from distinct source recordings with different lengths.

### The discovery: MP3 frame quantization

We cross‑referenced the source audio with the ElevenLabs output and found the culprit.  ElevenLabs generates MP3s in discrete chunks called *frames*.  If two original verses have similar lengths—within approximately 0.026 seconds of each other—the MP3 encoder rounds them to the exact same number of frames.  That produces identical file sizes and identical duration floats (e.g. `10.448937`), which in turn triggers the `ffmpeg-python` hash collision.

The durations weren't just *similar*—they were mathematically identical down to the floating‑point representation, because they came from the same underlying frame count.

### The cleanup: decoupling streams

While hunting the bug, we also simplified the rendering logic in `ParashahEpisodeVideo` and `ParashahVideo`.  The complex `segments` property was replaced with independent `video_stream` and `audio_stream` properties, separating concerns cleanly.  We realised that since ElevenLabs provides the correct length, we could remove the `.audio.filter('atrim')` call entirely—FFmpeg simply reads until End‑of‑File.  Unnecessary safety checks were commented out, making the code more readable and maintainable.

### The solution: global cache busting

The fix moved out of the video logic entirely and into the data layer within `AudioAI.py`.  When the class loads durations from `cloned.csv` into memory, it now tracks every value in a set.  If it encounters a duplicate duration, it subtracts a single microsecond (`0.000001`) until the value is unique.

Why this works:
* **Mathematically unique** — every duration in memory is now a distinct value, so the library never attempts to merge nodes.
* **Physically imperceptible** — a microsecond difference is invisible to the human eye and completely inaudible.
* **Non‑destructive** — the actual `cloned.csv` on disk remains unchanged and accurate; only the in‑memory representation is adjusted.

With this fix, we can render anything from a single episode to the entire Bible without a single graph conflict.  The bug had been a blocker for months; the solution was one line in the cache loader.
