# 9 Oct 2025 – DeepSeek 3 – Unified presentation layer with Word class

_Media, Code_

The platform had a problem that was invisible until you looked closely at the rendered output.  Presentation logic was scattered across Flask routes, Parashah episodes all displayed Episode 1 content regardless of what you clicked, and there was no single, reliable way to turn content into something visual — whether that meant HTML, an image, or a PDF.

We fixed it by creating a unified presentation layer anchored by a new `Word` class in `Overlay.py`.  Every content object now exposes a `.presentation` property with an identical three‑level structure: **Paragraphs → Lines → Words**.  The `Word` objects carry verse context, so renderers can number things intelligently without parsing the source text again.

### Content class unification

All three content types now look the same under the hood:
* `Psalm` gained a paragraph structure on top of its existing poetic format.
* `Article` and `ParashahEpisode` share the same narration‑formatting logic.
* The episode‑distribution bug in `BibleParashot.py` — every episode showing episode 1 — was finally squashed.

### Visual rendering

The `.presentation` property became the single source of truth.  Flask templates, image generation, and PDF export all consume exactly the same data structure.  Templates now *only* consume presentation output; they contain no parsing logic.  The styling is minimal, semantic, and CSS‑driven.

The data flow is now a straight line:

```
Markdown Files → Text Parsing → Content Classes → Presentation Properties → Renderers
```

### What we fixed

* Episode content distribution — all episodes were showing the same content, now fixed.
* Markdown parsing inconsistencies — the rules are applied uniformly regardless of content type.
* Verse numbering in continuous narration flow — `Word` objects carry their verse references, so smart numbering works everywhere.

### The outcome

A production‑ready rendering system that handles Psalms (poem style with verse breaks), Articles (narration style with continuous flow), and ParashahEpisodes (same narration style) through identical pipelines.  The architecture is extensible — any future output format just needs to consume the `.presentation` property.  The bugs are fixed, the structure is consistent, and the API is simple enough that a developer can understand it in minutes.
